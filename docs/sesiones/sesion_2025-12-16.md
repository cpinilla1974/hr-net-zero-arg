# Sesión de Trabajo - Sistema de Seguimiento con Carga de Archivos

**Fecha**: 16 de Diciembre 2025
**Objetivo**: Implementar el sistema de seguimiento MRV basado en carga de archivos

---

## Contexto

Basándome en la documentación de la reunión del 13/dic/2025, se identificó que el sistema de seguimiento NO son dos procesos separados, sino **UN único proceso de reporte anual estructurado en capas incrementales**:

### Estructura del Reporte Anual

```
Formulario Único Anual [Año]
├── Archivo GNR (Getting the Numbers Right)     - Base común obligatoria
├── Archivo Adicionales 2030                     - Indicadores emergentes obligatorios
└── Archivo Adicionales 2050                     - Indicadores largo plazo opcionales
```

### Diferencia de Granularidad

| Capa | Nivel de reporte |
|------|------------------|
| GNR + Adicionales 2030 | Por planta |
| Adicionales 2050 | Consolidado por empresa |

---

## Decisión de Diseño: Archivos en lugar de Formularios

**Decisión tomada**: En lugar de crear formularios web complejos, se optó por un sistema de **carga de archivos Excel** con plantillas predefinidas.

**Razones**:
1. Empresas ya están familiarizadas con el protocolo GNR en formato Excel
2. Más rápido de implementar y mantener
3. Permite validación offline antes de subir
4. Facilita el procesamiento batch por parte de supervisores
5. Compatible con flujos de trabajo existentes en las empresas

## Flujo de Usuario

1. **Informante de Empresa**:
   - Selecciona año del reporte
   - Descarga plantilla GNR
   - Completa datos de todas las plantas
   - Sube archivo GNR
   - Descarga plantilla Adicionales 2030
   - Completa indicadores emergentes (hidrógeno, etc.)
   - Sube archivo Adicionales 2030
   - (Opcional) Descarga plantilla Adicionales 2050
   - (Opcional) Completa indicadores consolidados empresa
   - (Opcional) Sube archivo Adicionales 2050
   - Agrega observaciones si es necesario
   - Envía a supervisor de empresa

2. **Supervisor de Empresa** (próxima implementación):
   - Revisa archivos cargados
   - Valida datos
   - Aprueba o rechaza con comentarios

3. **Coordinador País / Admin** (próxima implementación):
   - Monitorea progreso de empresas
   - Agrega datos a nivel nacional

### Workflow Configurado (4 pasos)

El proceso define un flujo simplificado de 4 pasos:

#### 1. BORRADOR
- **Roles**: INFORMANTE_EMPRESA
- **Descripción**: Informante cargando archivos en borrador
- **Visible**: No (estado interno)

#### 2. ENVIADO
- **Roles**: INFORMANTE_EMPRESA
- **Descripción**: Enviado a FICEM para procesamiento
- **Notifica a**: ADMIN_PROCESO

#### 3. EN_REVISION_FICEM
- **Roles**: ADMIN_PROCESO
- **Descripción**: FICEM procesando y validando datos

#### 4. APROBADO_FICEM
- **Roles**: ADMIN_PROCESO
- **Descripción**: Datos procesados y agregados calculados
- **Triggers**:
  - `calcular_agregado_nacional`
  - `generar_trayectorias`

### Hojas Requeridas

El proceso espera tres archivos Excel:
1. **GNR**: Datos base Getting the Numbers Right
2. **Adicionales_2030**: Indicadores emergentes
3. **Adicionales_2050**: Indicadores largo plazo

### Cálculos Habilitados

- `gcca`: Cálculos del Global Cement and Concrete Association
- `bandas`: Análisis de bandas de desempeño
- `trayectorias`: Generación de trayectorias 2030/2050
- `agregado_nacional`: Consolidado a nivel país

### Problema Técnico Resuelto

Durante la creación del proceso se encontró una inconsistencia entre:
- **Database CHECK constraint**: Requiere valores lowercase ('activo', 'borrador', etc.)
- **SQLAlchemy Enum**: Define valores uppercase (ACTIVO, BORRADOR, etc.)

**Solución aplicada**: Usar INSERT SQL directo con valores lowercase para evitar la conversión automática del enum.

**Implicación**: El ORM de SQLAlchemy no puede leer este registro directamente. Las consultas deben hacerse con SQL raw o actualizando el modelo para aceptar lowercase.
